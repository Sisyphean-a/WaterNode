# WaterNode 工具化重构交接设计文档

## 1. 文档目的

本文档用于将 WaterNode 当前的重构诉求整理为一份可持续迭代的交接设计文档，供 `CodexPotter` 在后续独立上下文中继续执行。本文档只描述目标结果、约束、接口事实、验收标准与禁止退化项，不给出具体代码实现步骤。

## 2. 背景

WaterNode 当前已经完成过一轮“工具工作台化”重构，但仍存在以下关键问题：

1. 账号体系命名混乱，仍有“凭证”“数据源”等歧义概念。
2. 首页仍然不是纯工具工作台，信息噪音过多，主任务不突出。
3. 取水流程没有充分围绕“选择账号 -> 选择区域 -> 立即取水”展开。
4. 侧边栏交互仍不够自然，点击后不会自动折叠。
5. 签到、余额、账单的状态确认链路不完整，无法可靠判断是否成功。
6. 取水规格只暴露了 `15L`，但真实业务支持 `7.5L` 与 `15L` 两档。
7. 日志没有完全独立成“结果追踪页”，首页仍承担过多辅助信息展示职责。

本次任务允许破坏性重构，目标不是在旧结构上修修补补，而是把项目重构为更符合“高频工具”标准的形态。

## 3. 当前问题摘要

### 3.1 账号语义不清

- UI 与代码中存在“凭证管理”“数据源”等命名。
- 从用户视角，这些内容本质上都是“账号”。
- 当前取水失败时，错误会直接暴露为 `AppException(message: 超出每日取水【2】次数,请明日再试)`，但产品层没有给出“切换账号继续操作”的自然路径。

### 3.2 首页信息架构不合理

- 首页有过多描述性内容、日志、快捷操作和稀疏留白。
- 首页没有把“取水”作为第一优先级任务。
- 顶部统计项包含“在线账号”“失效账号”等非核心信息。

### 3.3 导航与页面边界不合理

- 终端大厅和批量操作仍是独立页面，但从业务视角看，它们都应收敛到首页工作台。
- 日志应独立成页，而不是长期占据首页空间。
- 侧边栏点击后不自动折叠，尤其在窄屏和抽屉场景下交互体验差。

### 3.4 结果确认不足

- 签到无法明确得知“未签到 / 已签到 / 本次签到成功 / 失败”。
- 余额与账单虽然已有部分能力，但尚未形成统一、可验证、可追溯的状态链路。

## 4. 重构目标

本次重构的目标是把 WaterNode 改造成一个以“取水”为中心的紧凑工具台：

1. 首页只承担核心任务入口与核心统计展示。
2. 账号相关概念全部收敛为“账号管理”。
3. 取水操作必须天然支持多账号、多区域、两种水量规格。
4. 任意关键动作都要有可确认的状态反馈与日志追踪。
5. 侧边栏与主体区的关系要符合工具型应用的常见交互，不制造整页重建或明显割裂感。

## 5. 范围

### 5.1 本次必须覆盖

- 首页重构为“取水工作台”
- 批量操作能力下沉到首页底部
- 终端大厅能力合并到首页
- 日志独立成页
- “凭证管理”更名为“账号管理”
- 账号支持备注
- 账号可切换，且取水流程可明确使用指定账号
- 首页默认账号选择策略与默认区域策略
- 取水支持 `7.5L` 和 `15L`
- 余额、签到状态、账单接口接入并形成可验证状态链
- 侧边栏点击后的自动折叠
- 现有冗余模块和旧信息结构的移除

### 5.2 本次不要求覆盖

- 后端接口设计调整
- 登录协议本身的改造
- 新增新的营销活动玩法
- 改造为 Web 端
- 引入新的状态管理框架

## 6. 术语统一

以下术语必须统一，文档、UI、代码命名都要围绕统一概念表达：

| 当前概念 | 目标概念 | 说明 |
|---|---|---|
| 凭证管理 | 账号管理 | 用户视角需要管理的是账号，不是技术性“凭证” |
| 数据源 | 账号 | 当前场景下不存在单独的数据源概念 |
| 终端大厅 | 首页取水工作台的一部分 | 不再作为独立核心工作页存在 |
| 批量操作页 | 首页底部批量操作区 | 不再作为独立核心工作页存在 |

禁止出现的结果：

- UI 中继续把账号称为“数据源”
- 交接文档和实现结果中继续把账号管理页当作“凭证列表页”

## 7. 目标产品结构

### 7.1 导航层级

重构后，产品应以“少页面、高聚焦”为原则。核心页面建议收敛为：

1. 首页工作台
2. 账号管理
3. 日志页
4. 登录授权页

说明：

- 首页工作台承担原“首页 + 终端大厅 + 批量操作页”的主要功能。
- 日志从首页剥离，变成独立结果追踪页。
- 登录授权页仍可存在，但其定位应从“大功能页”退回为“账号管理的附属流程页”。

### 7.2 首页工作台结构

首页应分为三个主要层次：

1. 顶部统计区
2. 中部核心取水区
3. 底部批量操作区

其中：

- 顶部统计区只保留“账号总数”和“总积分”两个指标。
- 中部核心取水区是首页第一视觉重点。
- 底部批量操作区用于承接原批量任务操作。

### 7.3 日志页结构

日志页用于集中展示以下结果：

- 取水结果
- 签到结果
- 抽奖结果
- 批量任务结果
- 错误堆栈或业务错误信息

首页不再承担完整日志展示职责。

## 8. 核心功能需求

### 8.1 账号管理

#### 目标

把当前“凭证管理”升级为真正的“账号管理”。

#### 必须满足

1. 页面名称改为“账号管理”。
2. 账号列表支持查看账号基础信息。
3. 每个账号支持设置备注。
4. 备注属于账号的一部分，应持久化保存。
5. 页面中移除“在线模块”这类独立展示区。
6. 账号列表本身应承担状态表达，不需要额外的大块状态面板。
7. 用户可以明确知道当前有哪些账号可选。

#### 行为要求

1. 账号备注可为空。
2. 若无备注，页面可以回退展示手机号或其他已有主标识。
3. 若一个账号已达到每日取水次数上限，不应让整个取水流程失效。
4. 必须允许切换账号后继续取水。

### 8.2 首页顶部统计区

#### 目标

首页顶部只展示两个最关键的总览指标。

#### 必须满足

1. 只显示“账号总数”。
2. 只显示“总积分”。
3. 两个指标必须并排展示。
4. 空间不足时优先缩小间距和留白，而不是换行堆叠。
5. 移除“在线账号”“失效账号”等非核心指标。

### 8.3 首页核心取水区

#### 目标

首页成为最直接的取水入口，不要求先进入终端大厅。

#### 必须满足

首页中部取水区拆分为三个小块：

1. 选择账号
2. 选择区域
3. 取水按钮区

#### 默认选择策略

1. 默认账号为“当前积分最高的账号”。
2. 默认区域为“该账号对应的默认区域”。

#### 取水按钮区要求

1. 必须同时提供 `7.5L` 与 `15L` 两个按钮。
2. 两个按钮属于同一条取水链路，只是水量档位不同。
3. 用户必须明确知道自己当前将使用哪个账号、哪个区域、哪种水量发起取水。

#### 业务约束

1. `7.5L` 与 `15L` 都是正式能力，不是隐藏能力。
2. 若账号当日取水次数用尽，应清晰提示该账号不可继续取水。
3. 若存在其他可用账号，应允许立即切换，不应把失败误导成“系统整体不可用”。

### 8.4 批量操作区

#### 目标

把“批量操作”收敛到首页底部，避免独立页面分散主任务。

#### 必须满足

1. 原批量任务页可被移除。
2. 批量签到、批量抽奖等批处理能力集中放到底部区域。
3. 该区域是辅助能力，不得抢占首页首屏核心视觉。

### 8.5 日志页

#### 目标

把日志从首页完全剥离，形成独立的结果追踪页。

#### 必须满足

1. 首页不再展示日志模块。
2. 日志页独立展示最近操作记录。
3. 用户可以从日志中分辨账号、动作、结果、错误信息。
4. 日志页应承担排错与结果追踪职责。

### 8.6 侧边栏交互

#### 目标

使侧边栏行为符合工具型应用的正常预期。

#### 必须满足

1. 点击侧边栏导航项后，主体内容切换。
2. 侧边栏在需要折叠的场景下应自动折叠。
3. 不允许出现“点击导航项后侧边栏仍异常占位”的结果。
4. 不允许因为切换页面导致整个应用壳层像重新刷新一样闪烁。

## 9. 签到、余额、账单三条状态链

### 9.1 余额链路

#### 目标

“剩余积分”必须使用专门的余额接口作为事实来源，不再依赖其他接口推断。

#### 事实来源

- 接口：`GET /pay/account/coin/user`
- 关键参数：`accountType=COIN`、`userId=<userId>`
- 关键字段：`data.totalFee`

#### 产品要求

1. 账号积分余额以该接口返回值为准。
2. 首页总积分、账号列表积分、默认账号排序都应基于这条链路。
3. 该接口返回成功后，页面相关积分展示应同步刷新。

### 9.2 签到状态链路

#### 目标

系统必须知道“今天到底是否已经签到”，不能只盲目发签到请求。

#### 事实来源

- 接口：`GET /marketing/userSgin/consSignDay`

#### 已知语义

- 返回 `ok=true` 可视为当前账号已签到。
- 响应中的 `data` 是连续签到天数等辅助信息，可作为展示数据，但“是否已签到”是更核心的判定目标。

#### 产品要求

1. 用户能够明确看到今日签到状态。
2. 若已经签到，不应继续表现为“未知”或“疑似失败”。
3. 签到操作结束后，状态应可验证。
4. 该状态要能进入日志。

### 9.3 账单链路

#### 目标

小康豆账单用于验证积分变化与业务动作是否落账。

#### 事实来源

- 接口：`GET /pay/user/accountDetail/bean/list`

#### 产品要求

1. 能查看最近账单。
2. 能分辨收入/支出。
3. 能看到账单类型、时间、备注、变化后总积分。
4. 对签到结果，能识别 `SIGN_IN`。
5. 对取水结果，能识别 `SCAN_FETCH_WATER`。

## 10. 接水业务需求

### 10.1 已知真实接口事实

来自现有接口文档与抓包记录：

1. 默认免费接水配置接口会返回：
   - 单次默认水量 `waterVolume=7.50`
   - 每日上限 `dayLimit=2`
2. 发起取水接口是：
   - `GET /marketing/app/freeWaterActivity/fetchWaterByScan?deviceId={deviceId}&num={num}`
3. 真实业务上：
   - `num=1` 对应 `7.5L`
   - `num=2` 对应 `15L`

### 10.2 产品解释要求

系统不应只暴露其中一种规格，更不应只默认展示 `15L`。

### 10.3 失败语义要求

当服务端返回：

- `超出每日取水【2】次数,请明日再试`

应视为“当前账号当日额度已耗尽”的业务失败，而不是网络失败或系统故障。

产品层必须体现：

1. 失败原因属于账号额度限制。
2. 该失败与设备列表加载失败、接口不可用、网络错误是不同类别。
3. 如果存在其他可用账号，用户有机会继续完成目标。

## 11. 接口事实汇总

本节用于把后续实现中必须依赖的接口事实集中固化，避免在聊天里反复口述。

### 11.1 公共说明

以下接口均来自当前已知真实环境。文档中不写真实敏感值，统一以占位符表示。

公共占位符：

- `<token>`：账号 Token
- `<userId>`：账号 userId
- `<deviceId>`：设备 ID
- `<pageNum>`：分页页码
- `<pageSize>`：分页大小

### 11.2 实际余额接口

#### 请求

```http
GET https://gateway.exiaokang.cn/pay/account/coin/user?accountType=COIN&userId=<userId>
Platform-Type: APPLETS
User-Id: <userId>
Device-Id: <deviceId>
Content-Type: application/json
Token: <token>
```

#### 成功响应示例

```json
{
  "code": "200",
  "msg": "用户金额查询成功",
  "data": {
    "totalFee": 10125.0
  },
  "ok": true
}
```

#### 事实要求

- 可用余额字段是 `data.totalFee`
- 页面看到的“剩余积分 / 总积分”都必须以它为准

### 11.3 签到状态校验接口

#### 请求

```http
GET https://gateway.exiaokang.cn/marketing/userSgin/consSignDay
Platform-Type: APPLETS
User-Id: <userId>
Device-Id: <deviceId>
Content-Type: application/json
Token: <token>
```

#### 响应示例

```json
{
  "code": "200",
  "msg": "操作成功",
  "data": 62,
  "ok": true
}
```

#### 事实要求

- 当前需求把 `ok=true` 视为“已经签到”
- 该接口是签到状态确认依据，不只是附加信息

#### 村内设备列表

```http
platform-type: CUSTOMER_APP
page-size: 10

```

### 11.7 取水接口

#### 请求

```http
GET https://gateway.exiaokang.cn/marketing/app/freeWaterActivity/fetchWaterByScan?deviceId=<deviceId>&num=<num>
platform-type: CUSTOMER_APP
token: <token>
```

#### 成功响应

```json
{"code":"200","msg":"操作成功","ok":true}
```

#### 失败响应示例

```json
{
  "status": 200,
  "code": "9999",
  "msg": "超出每日取水【2】次数,请明日再试",
  "error_code": "9999",
  "error_msg": "超出每日取水【2】次数,请明日再试"
}
```

#### 事实要求

- HTTP 200 不代表业务成功
- 必须解析响应体中的业务码和消息
- 该失败属于“每日次数耗尽”

## 12. 非目标与禁止走捷径

### 12.1 非目标

- 不要求为了完成重构而保留旧页面结构
- 不要求兼容“数据源”这一命名
- 不要求首页继续承载日志展示

### 12.2 禁止事项

1. 不得继续把首页做成信息看板式布局。
2. 不得继续保留“在线账号”“失效账号”两类首页指标。
3. 不得继续让“终端大厅”承担唯一取水入口。
4. 不得继续让“批量操作页”成为核心页面。
5. 不得继续让“日志”占据首页核心空间。
6. 不得把账号取水受限误报成系统异常。
7. 不得只支持 `15L` 而忽略 `7.5L`。
8. 不得继续使用“数据源”命名。

## 13. 风险

1. 页面边界变化较大，涉及导航、首页、账号管理、设备域、日志域的重新收敛。
2. 余额、签到状态、账单三条链路接入后，状态同步逻辑会比当前更复杂。
3. 账号备注、默认账号选择、默认区域选择都可能影响现有测试。
4. 如果继续沿用旧页面概念，会导致产品结构与需求目标互相冲突。

## 14. 验收标准

### 14.1 信息架构

- 首页只承担总览、取水、批量操作
- 日志独立成页
- 终端大厅不再是主取水入口
- 批量操作页不再是主功能入口

### 14.2 命名

- UI 中不再出现“凭证管理”“数据源”
- 统一收敛为“账号管理”“账号”

### 14.3 账号能力

- 账号可备注
- 账号可切换
- 账号积分可刷新并作为默认排序依据

### 14.4 取水能力

- 首页可直接选择账号和区域后发起取水
- 同时提供 `7.5L` 与 `15L`
- 当一个账号超额时，系统能明确表达为该账号额度耗尽

### 14.5 状态可验证性

- 用户能知道是否已签到
- 用户能查看最近账单并识别签到和取水记录
- 页面显示的积分与余额接口一致

### 14.6 交互

- 点击侧边栏后主体切换正常
- 侧边栏在应折叠场景下会自动折叠
- 不出现整页刷新感
