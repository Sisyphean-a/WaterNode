## 统一响应特征

成功响应主要有两种：

1. 简单成功：

```json
{"code":"200","msg":"操作成功","ok":true}
```

2. 带业务数据：

```json
{"code":"200","msg":"操作成功","data":{...},"ok":true}
```

失败响应样本：

```json
{
  "status": 200,
  "timestamp": "2026-04-15 19:44:05",
  "path": "",
  "method": "GET",
  "code": "9999",
  "msg": "超出每日取水【2】次数,请明日再试",
  "error_code": "9999",
  "error_msg": "超出每日取水【2】次数,请明日再试"
}
```

重点说明：

- HTTP 状态码仍然是 `200`
- 真正的业务成功失败要看响应体里的 `code`、`msg`、`ok` 或 `error_code`

## 重要调用逻辑

从时间顺序看，当前这组报文的主要业务链路如下：

1. 进入水机列表页，上报埋点 `WATER_PURIFIER_SCREEN`
2. 拉取默认接水配置 `findOneConfig`
3. 拉取默认分页设备列表 `listPage`
4. 拉取当前村设备列表 `list/inVillage`
5. 选择目标设备后，通过 `findByDeviceId` 或 `findByDeviceNum` 获取设备详情
6. 进入兑换页面，上报埋点 `FREE_WATER_EXCHANGE_SCREEN`
7. 调用 `fetchWaterByScan` 发起接水
8. 若接水成功，再上报埋点 `FREE_WATER_EXCHANGE_SUCCESS_SCREEN`

一个关键业务约束是：

- 是否能继续接水，不只取决于设备详情
- 更取决于 `findOneConfig` 返回的 `dayLimit` 和本日已成功接水次数

```json
}
```

### 失败根因

已知事实：

- 默认配置 `dayLimit=2`
- 前两次 `num=1` 都成功
- 第三次返回明确错误：`超出每日取水【2】次数,请明日再试`

可以得出结论：

- 第三次失败的直接原因是已经达到当日取水次数上限
- 不是网络失败，不是 HTTP 失败，也不是接口不存在
- 从响应文案看，服务端按“每日次数”做限制

进一步推断：

- 如果系统按成功次数累计，那么前两次成功已经用满 `2` 次额度
- 此时第三次无论是 `num=1` 还是 `num=2`，都可能被直接拒绝
- 如果系统同时按档位校验，那么 `num=2` 对应 `15L`，也会比单次 `7.5L` 占用更多额度

但仅凭当前报文，最稳妥的结论仍然是：

- 第三次失败的确定性原因是命中了 `dayLimit=2`

## 接水接口的最小调用链路

按你的要求，这里不给 `fetchWaterByScan` 发真实请求，只给出从拿到参数到最终调用所需的最小链路。

### 链路 A：直接接水所需最小参数链

1. 已有登录态 `token`
2. 调用 `GET /marketing/app/freeWaterActivityConfig/findOneConfig`

最小头：

```http
token: <token>
platform-type: CUSTOMER_APP
```

拿到关键字段：

- `data.waterVolume=7.50`
- `data.dayLimit=2`

作用：

- 确认单次默认水量
- 计算目标水量对应的 `num`

3. 调用 `GET /marketing/app/waterDispenser/list/inVillage`

最小头：

```http
token: <token>
platform-type: CUSTOMER_APP
page-size: 10
page-num: 0
```

拿到关键字段：

- `data.content[].id` -> 最终接水用的 `deviceId`
- `data.content[].deviceNum`

4. 最终准备调用：

```http
GET /marketing/app/freeWaterActivity/fetchWaterByScan?deviceId={deviceId}&num={num}
```

最终所需核心参数：

- 请求头：
  - `token`
  - `platform-type`
- 查询参数：
  - `deviceId`
  - `num`

其中：

- `deviceId` 来自设备列表
- `num` 由目标水量和 `waterVolume` 推断

根据已知样本：

- `waterVolume=7.50`
- `num=1` 对应 `7.5L`
- `num=2` 对应 `15L`

### 链路 B：带设备校验的最小链路

如果在最终接水前还想校验设备状态，最小链路如下：

1. `findOneConfig`
2. `list/inVillage` 或 `listPage`
3. `findByDeviceId?deviceId=...`
4. `fetchWaterByScan?deviceId=...&num=...`

其中第 3 步最小头：

```http
token: <token>
platform-type: CUSTOMER_APP
```

可拿到：

- `happyTiDeviceStatus`
- `happyTiDeviceStatusDesc`
- `signal`
- `tds`

### 当前最小链路结论

如果只看“拿到最终调用参数”这件事，最短链路是：

1. `findOneConfig` 拿 `waterVolume`
2. `list/inVillage` 拿 `deviceId`
3. 组装 `fetchWaterByScan?deviceId={deviceId}&num={num}`

最小必要信息只有两类：

- 认证信息：`token + platform-type`
- 业务参数：`deviceId + num`
